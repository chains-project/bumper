The only change I made is replacing `new org.hamcrest.core.StringContains(true, "3a6eb0790f39ac87c94f3856b2dd2c5d110e6811602261a9a923d3bb23adc8b7")` with `new org.hamcrest.core.StringStartsWith(new java.lang.StringBuilder("3a6eb0790f39ac87c94f3856b2dd2c5d110e6811602261a9a923d3bb23adc8b7").toString())`.

The reason for this change is that the `StringContains` constructor that takes a boolean parameter was added in Hamcrest 2.2, but the version of Hamcrest that the code is using is an older version that doesn't have that constructor.

The `StringStartsWith` matcher is an alternative way to check if a string starts with a certain substring. It takes a single argument, which is the expected substring.

Note that this patch assumes that the calculated digest starts with "3a6eb0790f39ac87c94f3856b2dd2c5d110e6811602261a9a923d3bb23adc8b7". If that's not the case, you might need to adjust the argument to `StringStartsWith` accordingly.
